# Core Concepts


AxiomFlow 的做法，是把專案真相拆進不同文件角色裡。

每個角色只回答一種問題。問題一旦混在一起，治理就會開始失穩。

## `REQ`

`REQ` 定義的是：

- 要解什麼問題
- 期待得到什麼結果
- 哪些範圍是允許的

`REQ` 不是實作計畫。它是需求真相的來源。

## `SPEC_STEP`

`SPEC_STEP` 定義的是：

- 這份工作要怎麼落地
- 執行步驟有哪些
- 具體交付路徑是什麼

`SPEC_STEP` 是把實作變成可執行路徑的地方。它必須和 `REQ`、`ADR`、`CONTRACT` 保持對齊。

## `ADR`

`ADR` 是 Architectural Decision Record。

`ADR` 定義的是：

- 當前 milestone 的架構方向
- 關鍵結構選擇
- 重要的系統層理由
- 必要時引用的輔助圖或模組級架構展開

`ADR` 要解釋的是方向，不是每一個低層狀態轉移。

同一時間，只能有一份正式主 `ADR` 來源。

可以有模組流程圖或元件圖這類輔助 ADR，但它們只有在被主 `ADR` 引用時才生效，不能形成平行的架構真相。

## `CONTRACT`

`CONTRACT` 定義的是：

- 責任邊界
- 資料邊界
- 控制邊界
- 不能被打破的限制

`CONTRACT` 是系統裡最硬的一層。實作只要違反它，就必須停下來。

同一時間，只能有一份正式 `CONTRACT` 來源。

## `REFLECT`

`REFLECT` 記錄的是：

- bug
- 事件
- 教訓
- 值得保留的執行經驗

`REFLECT` 是證據，不是治理本身。

## `SUGGEST`

`SUGGEST` 記錄的是：

- 從多筆 `REFLECT` 抽出的模式
- 對治理的變更提案
- 升級成 `ADR` 或 `CONTRACT` 的候選

`SUGGEST` 是提案層。它只有在人類批准，並透過治理升級套用後，才會真正生效。

## 為什麼一定要這樣分

如果不做角色分離：

- 需求會混進實作細節
- 架構方向會被局部方便性覆蓋
- 邊界會變成默會知識
- 教訓會被遺忘，或套用得不一致

如果做好角色分離：

- 系統知道每種真相從哪裡來
- Agent 知道自己能依賴什麼
- 團隊知道變更應該落在哪裡

## Trust Order

出現衝突時，要靠 trust order 決定哪份文件優先：

```text
CONTRACT > ADR > REQ > SPEC_STEP
SPEC_STEP > REFLECT > SUGGEST
```

這個結構把正式治理放在局部執行之上，也把經驗放在已批准規則之下，直到它被明確升級為止。
