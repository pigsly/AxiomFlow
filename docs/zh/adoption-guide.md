# Adoption Guide


這份文件講的是，怎麼把 AxiomFlow 帶進一個真實團隊，而不是一上來就重做整套管理。

## 先從小處開始

不要一開始就把它做成完整的文件官僚體系。

先建立那個最小、但足以擋下危險執行的治理面：

- 一份 `REQ`
- 一份 `SPEC`
- 一份 `ADR`
- 一份 `CONTRACT`

這已經足夠先立住：

- 要做什麼
- 要怎麼做
- 架構往哪裡走
- 哪些邊界不能踩

## Phase 1: 先建立控制，再談擴張

第一個目標不是完整。

第一個目標是可控。

在這個階段：

1. 定義一份當前有效的需求。
2. 定義一份當前可執行的規格。
3. 定義一份當前架構方向。
4. 定義一份當前邊界契約。
5. 要求實作前必須先做 `PDR`。

只要做到這一步，你的 AI 交付迴路就已經比大多數團隊安全。

## Phase 2: 把停機條件立起來

最小文件組立住之後，就要開始執行停機規則。

團隊必須先同意，只要發生以下任何一種情況，執行就停：

- `SPEC` 與 `REQ` 衝突
- 實作方向與 `ADR` 衝突
- 提議中的工作違反 `CONTRACT`
- 關鍵資訊缺失

從這一刻開始，這套系統改變的就不只是文件，而是行為。

## Phase 3: 把學習留下來

當團隊已經穩定使用 `PDR`，就開始透過 `REFLECT` 留下有意義的經驗。

不要什麼都記。

只記那些真的值得保留的：

- 重複出現的誤解
- 因文件失配造成的 bug
- 一再出現的邊界違反
- 應該影響未來治理的執行模式

## Phase 4: 有意識地升級治理

當模式開始重複，就建立 `SUGGEST`。

這一步，是局部經驗走向正式治理的橋。

治理升級只能透過人類批准後的 `GU`。

這很重要，因為它避免系統每學到一件新事就立刻失穩。

## 團隊角色

不同團隊可以用不同方式分配角色，但這些責任一定要有人承擔：

- 有人負責需求真相
- 有人負責架構方向
- 有人負責硬邊界定義
- 有人執行或批准 pre-development review
- 有人決定提案是否成為正式治理

這個模型不要求僵硬的組織圖，但要求清楚的決策責任。

## 常見導入錯誤

避免這些錯法：

- 還沒定權責，就先寫一堆模板
- 把 `ADR` 當成實作細節
- 把 `CONTRACT` 寫成長篇技術備忘，而不是硬邊界
- 把每一件事都記進 `REFLECT`
- 還沒批准，就讓 `SUGGEST` 像正式治理一樣生效
- 衝突已經看見了，卻還讓實作繼續

## 什麼叫導入得好

當你看到這些狀態時，代表導入方向是對的：

- Agent 不會在對齊檢查前就開工
- 衝突會導向升級，而不是即興處理
- 架構與邊界在交付過程中始終可見
- 教訓會被保存，但不會繞過正式治理
- 團隊能說清楚某個決策為什麼被允許、阻斷或升級

## 建議導入順序

對大多數團隊來說，這個順序就夠：

1. 發布 `README.md`
2. 建立最小文件組
3. 要求 `PDR`
4. 套用停機規則
5. 開始使用模板
6. 記錄有意義的 `REFLECT`
7. 用 `SUGGEST` 檢查重複模式
8. 透過已批准的 `GU` 升級治理
