# 怎麼跟 Codex 合作


這份文件不講治理模型本身。

它只回答一個問題：

**你要怎麼跟 Codex 開口，才能讓它做對事。**

## 先記住一件事

Codex 不是讀心，也不是只靠一句話就能穩定做對。

你給得越清楚，它做得越準。

真正要先講清楚的，通常只有四件事：

- 你要它做什麼
- 範圍到哪裡
- 哪些檔案要先看
- 哪些邊界不能踩

## 最小起手式

如果你是第一次下指令，先用這個格式就夠：

```text
先讀 README.md、Governance.md、docs/getting-started.md。
我要你處理 ________。
範圍只到 ________。
不要改 ________。
開始前先做 PDR，如果文件衝突就停下來告訴我。
```

這種開法的重點不是客氣，而是先把：

- 任務
- 範圍
- 限制
- 停機條件

一次講清楚。

## 你至少要給什麼

### 1. 任務

不要只說：

```text
幫我改一下
```

要說成：

```text
幫我把登入流程改成 email login 優先，保留原本 OAuth。
```

### 2. 範圍

不要讓 Codex 自己猜它可以改到哪裡。

例如：

```text
只改 login flow 與相關文件，不要動 billing、permission。
```

### 3. 相關檔案

如果你知道哪些文件或程式最關鍵，直接先點名。

例如：

```text
先看 docs/REQ/REQ-003.md、docs/SPEC_STEP/SPEC-014.md、ADR.md。
```

### 4. 邊界或限制

如果有不能破壞的地方，要直接說。

例如：

```text
不要改 public API。
不要改資料表 schema。
不要繞過 CONTRACT。
```

## 什麼時候要要求它先停

以下情況，不要叫 Codex 直接做到底，而是要先要求它停下來回報：

- 文件彼此衝突
- `REQ` 和 `SPEC_STEP` 對不起來
- `ADR` 與現在做法明顯不一致
- `CONTRACT` 可能被踩
- 需求其實還沒講清楚

你可以直接這樣下：

```text
先做 PDR。
如果文件衝突、邊界不清或需求不完整，不要硬做，先停下來告訴我。
```

## 什麼任務可以直接下

以下任務，通常可以直接叫 Codex 開始做：

- 已經有明確 `REQ` / `SPEC_STEP` 的功能實作
- 範圍清楚的 bug 修正
- 已知檔案範圍內的重構
- 文件語氣整理、結構整理、連結補全
- 已經講清楚邊界的小型調整

這些情況的共通點是：

- 任務清楚
- 範圍可控
- 風險邊界已知

## 什麼任務不要直接下

以下任務，不適合一開始就叫 Codex 直接做到底：

- 需求還沒定
- `REQ` 和 `SPEC_STEP` 還沒對齊
- 你自己也不確定該改哪個模組
- 可能會踩到 `ADR` 或 `CONTRACT`
- 涉及跨模組重寫，但邊界還沒談清楚

這種情況比較好的開法是：

```text
先不要改。
先讀 ________。
幫我判斷這件事應該停在 REQ / SPEC_STEP，還是要升到 ADR / CONTRACT。
如果文件衝突或邊界不清，先列出問題。
```

也就是先讓它做判斷，再決定要不要進入實作。

## 四種最常用的指令開法

### 1. 叫它先讀，再做

```text
先讀 README.md、Governance.md、docs/concepts.md。
我要你根據 docs/REQ/REQ-002.md 和 docs/SPEC_STEP/SPEC-008.md 實作。
開始前先做 PDR，有衝突就停。
```

### 2. 叫它先找，再提方案

```text
先讀相關文件。
幫我找目前 notification flow 的實作入口、相關 ADR 與 CONTRACT。
先不要改 code，先告訴我你打算怎麼改。
```

### 3. 叫它直接改，但限制範圍

```text
修掉 export CSV 的 bug。
只改 export module 和相關測試。
不要動 API schema。
開始前先做 PDR。
```

### 4. 叫它做文件整理

```text
先讀 README.md、docs/project-scale.md、docs/getting-started.md。
幫我把語氣統一成現在 README 的風格。
不要改結構，只修字句與明顯重複。
```

## 什麼叫好指令

好指令通常有這些特徵：

- 有明確目標
- 有清楚範圍
- 有指定入口文件
- 有明確限制
- 有停機條件

例如：

```text
先讀 docs/REQ/REQ-005.md、docs/SPEC_STEP/SPEC-021.md、CONTRACT.md。
我要你完成 payment retry flow。
只改 payment 模組、相關測試與必要文件。
不要改 DB schema。
先做 PDR；如果需求、規格或邊界有衝突，就停下來列出問題。
```

## 什麼叫壞指令

壞指令通常不是語氣差，而是資訊不夠。

例如：

```text
幫我把這個系統整理一下。
```

```text
把這邊改好。
```

```text
你看一下哪裡怪怪的，順手一起修。
```

這種說法最大的問題，是：

- 任務不明
- 範圍不明
- 風險不明
- 停機條件不明

最後很容易讓 Codex 做出看起來積極、實際上越改越散的東西。

## 如果你趕時間

最短可以只講這五行：

```text
先讀 ________。
我要你做 ________。
只改 ________。
不要動 ________。
先做 PDR，有衝突就停。
```

這通常就已經比大多數模糊指令安全很多。

## 一句話原則

**不要只告訴 Codex「做什麼」，還要告訴它「做到哪裡、不能碰哪裡、什麼情況要停」。**
