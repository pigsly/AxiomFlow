# Upgrade Signals


這份文件不是正式規則。

它只回答一個很實際的問題：

**你怎麼知道自己該從簡單版升到普通版、進階版，甚至專業版？**

## 一個故事

一開始，團隊只是想讓 Agent 幫忙加快開發。

最早的需求都很單純。
改一個欄位，補一個 API，調整一個流程節點。

這時只要 `REQ`、`SPEC_STEP`、`ADR`、`CONTRACT` 對齊，事情通常就能安全完成。

這是[簡單版 README](../en/README.simple.md)想解的問題。

但系統不會永遠停在這裡。

## 訊號 1：需求不再是單點修改

原本只是改一個欄位、一個 API、一個流程節點。

後來開始變成：

- 改 A 要一起改 B、C、D
- 一個需求同時牽動資料、流程、權限、UI
- 改小地方會影響舊功能

這表示系統已從局部風險進入結構風險。

這時你至少要先判斷，自己還能不能停留在簡單版。

如果 `REQ + SPEC_STEP + PDR` 還足以控制風險，可以先不升。
如果開始反覆補洞，通常就會往普通版走。

## 訊號 2：同樣問題反覆出現

例如：

- 相同 bug 一直回來
- 同樣的邊界錯誤一直發生
- 每次都靠人記得，不是靠系統保證

這時就不能只修 code。

你需要開始把經驗留下來，讓錯誤不只是被修掉，而是被記住。

這就是[普通版 README](../en/README.standard.md)的起點：

- `REQ -> SPEC_STEP -> PDR -> WC -> REFLECT`

重點不是增加流程感，而是讓團隊不再只靠人腦保存教訓。

## 訊號 3：文件與實作開始分離

例如：

- `REQ` 說一套，`SPEC_STEP` 做另一套
- `ADR` 沒更新，但實際方向已經偏移
- `CONTRACT` 沒寫清楚，大家各自理解

這就是 `PDR` 真正要處理的場景：

在實作前，先檢查 `REQ / SPEC_STEP / ADR / CONTRACT` 是否還對齊。

如果這種分離已經不只是一兩次，而是多個事件反覆指向同一個結構問題，就不該只停留在普通版。

這時通常會進入[進階版 README](../en/README.advanced.md)，開始思考：

- 這是不是該升級成 `ADR`
- 還是應該升級成 `CONTRACT`

## 訊號 4：需要靠特定人才知道怎麼改

只要出現這種情況：

- 只有某個人知道這裡能不能動
- 只有某個人知道當年為什麼這樣做
- 一旦那個人不在，風險暴增

這表示知識還停在人身上，沒有沉澱成治理資產。

這通常代表你已經從普通版往進階版移動，因為問題不再只是要不要記錄，而是要不要把知識升成正式治理依據。

## 訊號 5：改動需要先停下來討論

只要開始常常出現這種情況：

- 這次改法會不會違反邊界？
- 這需求到底算不算原範圍？
- 這種例外以後還要不要繼續接受？

這代表你已進入需要治理模組參與決策的階段，而不再只是 coding。

這就是[專業版 README](../en/README.professional.md)處理的場景：

- 人類批准
- 正式治理升級
- 衝突阻斷
- 停機等待

這一層處理的已經不是實作細節，而是規則本身。

## 五個訊號對應哪一版

- 只是在做局部修改：停在[簡單版](../en/README.simple.md)
- 同樣問題反覆出現：升到[普通版](../en/README.standard.md)
- 文件與實作開始分離：通常要往[普通版](../en/README.standard.md)或[進階版](../en/README.advanced.md)移動
- 知識掌握在特定人身上：從[普通版](../en/README.standard.md)往[進階版](../en/README.advanced.md)
- 改動常常需要先停下來討論：進入[專業版](../en/README.professional.md)

## 讀法建議

如果你還不確定自己在哪一版：

1. 先讀這份故事
2. 再看[Version Guide](./project-scale.md)
3. 然後選一份對應版本 README
