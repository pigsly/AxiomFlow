# Conflict Handling


當治理真相變得不可靠，系統就必須停下來。

這是 AxiomFlow 最重要的規則之一。

## 什麼時候要啟動 Conflict Handling

只要你發現以下任一情況，就必須啟動 conflict handling：

- 文件彼此矛盾
- 實際邏輯已偏離宣告方向
- 關鍵資訊缺失
- 治理文件已經過期
- 輸出很可能違反 `ADR` 或 `CONTRACT`

## 強制回應

一旦偵測到衝突，就按這個順序處理：

1. 停止當前實作或文件更新。
2. 找出衝突來源。
3. 說清楚具體矛盾點。
4. 說明可能造成的影響。
5. 把決策交還給人類。

系統不能靠假設繼續往下做。

如果衝突是在 `PDR` 階段被發現，系統在把決策交還給人類之前，仍然應該先產出一份可讀結果。

這份結果至少要讓三件事變得可見：

- 整體訊號是什麼
- 衝突出現在哪裡
- 為什麼執行不能繼續

實際上，這通常會表現為：

- 一份 `PDR Summary`
- 一個 terminal status layout
- 一份簡短的 `Summary List`

## `SPEC` Blocking Rule

如果衝突出現在 `SPEC-XXX.md`，就把它的 `code_implement` 設為 `blocked`。

在以下條件成立之前，不得解除：

- 衝突文件已更新，或
- 人類已明確解決衝突

## 非 `SPEC` 的治理文件

如果衝突出現在正式治理文件裡，例如：

- `REQ`
- `ADR`
- `CONTRACT`

不要自行發明新欄位，也不要繞過問題硬補。

記錄警告，然後停下來。

## Conflict Handling 應該讓什麼變明確

Conflict handling 不只是停下來而已。

它還要把「為什麼停」這件事變得可判讀。

至少應該讓 reviewer 看見：

- 哪些文件彼此衝突
- 具體矛盾點是什麼
- 這是 `REQ` 衝突、`ADR` 衝突、`CONTRACT` 衝突，還是資訊缺漏問題
- 在重新進入 `WC` 前，什麼東西必須先被修正

## 為什麼系統必須停

很多團隊把停機條件當成摩擦。

這套系統把停機條件當成保護。

當真相不一致時，Agent 如果還繼續往前做，速度就會從優勢變成風險。它做得越快，失配就擴散得越快。

Conflict handling 的存在，是為了保住整個系統輸出的可信度。
