# Feedback Loop


AxiomFlow 不只是文件模型。

它是一個讓執行可控、讓學習可控的閉環。

沒有這個閉環，系統只會變成靜態文件。

有了這個閉環，系統才會變成治理引擎。

## 基本閉環

最小閉環長這樣：

```text
REQ -> SPEC -> PDR -> WC -> REFLECT -> SUGGEST -> GU -> ADR / CONTRACT
```

每一步各有自己的工作：

- `REQ` 定義要解什麼問題
- `SPEC` 定義工作要怎麼做
- `PDR` 在執行前檢查計畫是否對齊
- `WC` 只在對齊確認後才進行實作
- `REFLECT` 記錄執行中有意義的經驗
- `SUGGEST` 把重複教訓轉成治理提案
- `GU` 把已批准的提案升級成正式治理

這個閉環重要，是因為它能擋住兩種失敗模式：

- 執行在未對齊時就先發生
- 學習發生了，結構卻沒有跟著改善

## 為什麼這個環一定要閉合

很多團隊其實已經做了其中一部分：

- 寫需求
- 寫規格
- 做實作
- 事後學到一些事

但通常缺的是回流路徑。

教訓留在聊天記錄、ticket 註解，或個人記憶裡，卻沒有真的進入架構方向或邊界治理。

這套系統把經驗往上走的路補上了，但只能透過明確批准。

## 控制機制

這個閉環不只是讓事情流動，而是讓事情受控地流動。

讓這個閉環可信的，是三條規則：

- 對齊審查前，不得實作
- 衝突可見後，不得繼續
- 治理變更前，必須批准

正是這三條規則，阻止系統退化成即興應付。

## 回饋機制

回饋機制從 `REFLECT` 開始。

不是每一個事件都要變成治理。系統只該記住真正有意義的事，例如：

- 重複出現的執行失誤
- 因邊界混亂造成的 bug
- 從需求到規格的漂移
- 對架構的誤解
- 一再拖慢或傷害交付的治理缺口

一旦模式浮現，`SUGGEST` 就把它翻成治理提案。

關鍵差別在這裡：

- `REFLECT` 保存證據
- `SUGGEST` 詮釋證據
- `GU` 改變系統

這種分離，讓回饋閉環既有用，又不會讓治理失穩。

## 快迴路與慢迴路

這個模型裡，其實有兩個不同速度的迴路。

### 快迴路

快迴路處理的是日常交付：

```text
REQ -> SPEC -> PDR -> WC
```

它的工作，是控制執行品質。

### 慢迴路

慢迴路處理的是治理演化：

```text
REFLECT -> SUGGEST -> GU -> ADR / CONTRACT
```

它的工作，是改善系統本身的規則。

健康的專案會同時跑兩個迴路。不健康的專案只跑快迴路，然後反覆撞上同一種失敗模式。

## 好的回饋長什麼樣

好的回饋閉環，不會為每一件小事都製造文書負擔。

好的回饋閉環會做到：

- 在實作前抓到真正的失配
- 只留下有意義的教訓
- 辨認模式，而不是追逐雜訊
- 只有在人類確認模式成立後，才升級治理

這樣，系統才能同時保持嚴格與務實。
